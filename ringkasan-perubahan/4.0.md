```
GET   /admin                      ── auth + role:admin ── AdminController@index
POST  /admin/voucher/verify       ── auth + role:admin ── AdminController@verifyVoucher
PATCH /admin/voucher/{id}/complete ── auth + role:admin ── AdminController@completeRedeem
```

Seluruh route admin berada di dalam group `prefix('admin')->middleware('role:admin')`, yang sendiri sudah berada di dalam group `middleware('auth')`. Artinya diperlukan **dua layer otorisasi** sebelum akses diberikan:
1. User harus terautentikasi (session valid)
2. User harus memiliki `role = 'admin'`

## Mekanisme Keamanan Otorisasi State

### Mengapa Transisi `pending → completed` Memerlukan Database Transaction?

```php
$redeem = DB::transaction(function () use ($id): RewardRedeem {
    $redeem = RewardRedeem::lockForUpdate()->findOrFail($id);

    if ($redeem->status !== 'pending') {
        throw new \DomainException('Voucher ini sudah berstatus: ' . $redeem->status);
    }

    $redeem->update(['status' => 'completed']);
    return $redeem;
});
```

#### Problem: Race Condition pada Concurrent Request

Tanpa locking, skenario berikut bisa terjadi:

```
T1: Admin A baca status = 'pending'     ✓
T2: Admin B baca status = 'pending'     ✓ (belum berubah)
T3: Admin A update → 'completed'        ✓
T4: Admin B update → 'completed'        ✓ (DUPLIKAT!)
```

Dua admin bisa saja memverifikasi voucher yang sama secara bersamaan — warga menerima hadiah dua kali.

#### Solution: Pessimistic Locking + Transaction

```
T1: Admin A → lockForUpdate() → LOCK acquired, baca status = 'pending'
T2: Admin B → lockForUpdate() → BLOCKED (menunggu lock T1 release)
T3: Admin A → update 'completed' → COMMIT → lock released
T4: Admin B → lock acquired → baca status = 'completed' → DomainException!
```

`lockForUpdate()` mengunci row di level database (MySQL `SELECT ... FOR UPDATE`). Selama transaksi T1 berlangsung, transaksi T2 akan **menunggu** — bukan membaca data stale. Setelah T1 selesai, T2 membaca status terbaru (`completed`) dan langsung melempar exception.

Ini memastikan:
- **Atomicity**: Perubahan status terjadi sebagai satu unit kerja
- **Isolation**: Tidak ada dua transaksi yang bisa membaca status `pending` secara bersamaan
- **Idempotency Protection**: Voucher yang sudah `completed` tidak bisa di-complete ulang

#### Mengapa Tidak Cukup Hanya `if ($redeem->status !== 'pending')`?

Tanpa `lockForUpdate()`, pengecekan `if` dan `update` berjalan sebagai dua operasi terpisah. Di antara keduanya, proses lain bisa mengubah data — ini disebut **TOCTOU (Time of Check to Time of Use)** vulnerability. Pessimistic locking mengeliminasi celah ini sepenuhnya.

## Perbaikan Middleware `EnsureUserHasRole`

Middleware sekarang mendukung dual-response:

```php
if ($request->expectsJson()) {
    return response()->json([...], 403);
}

abort(403, 'Akses ditolak.');
```

- **API request** (header `Accept: application/json`): Response JSON 403
- **Web request** (browser): Laravel abort 403 → menampilkan halaman error standar

Perubahan ini backward-compatible — test API yang menggunakan `Accept: application/json` tetap menerima response JSON seperti sebelumnya.

## Keamanan Data Visual pada Portal Admin

### 1. XSS Prevention

Semua data dinamis dari response AJAX disisipkan via `textContent`:

```javascript
document.getElementById('detail-code').textContent = data.redemption_code;
document.getElementById('detail-user').textContent = data.user_name;
```

Tidak ada satu pun data API yang masuk melalui `innerHTML` — menutup vektor DOM-based XSS.

### 2. Server-Side Authorization Stack

```
Request → auth middleware → role:admin middleware → Controller
```

Tiga lapis sebelum kode controller dieksekusi. User biasa (`role: 'user'`) yang mencoba mengakses `/admin` akan mendapat abort 403.
